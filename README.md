# Дашборд «Victory парсинг» — автообновление из Bitrix24

Дважды в сутки (06:00 и 18:00 по Москве) скрипт заходит в Bitrix24, забирает лиды
и сделки с источником **«Victory парсинг»**, раскладывает их по дням и городам и
сохраняет в `data/months/*.json`. Дашборд (`dashboard/index.html`) читает эти файлы,
позволяет выбрать период/город и сам считает все показатели.

Никакого постоянно работающего сервера не нужно — GitHub бесплатно запускает
задачу по расписанию (GitHub Actions).

## Шаг 1. Входящий вебхук в Bitrix24

1. **Настройки → Разработчикам → Другое → Входящий вебхук**
2. Права минимум на: `crm` (или отдельно `crm.lead`, `crm.deal`, `crm.status`)
3. Скопируйте URL вида `https://ваш-домен.bitrix24.ru/rest/1/xxxxxxxxxxxxxxxxx/`
4. Никому не показывайте этот URL — по нему можно читать данные CRM

## Шаг 2. Репозиторий на GitHub

1. Заведите аккаунт на [github.com](https://github.com), если ещё нет
2. Создайте репозиторий, загрузите туда всё содержимое этой папки целиком,
   сохранив структуру (`.github/`, `scripts/`, `data/`, `dashboard/`, `package.json`)

## Шаг 3. Секрет с вебхуком

**Settings → Secrets and variables → Actions → New repository secret**
Имя: `BITRIX_WEBHOOK_URL`, значение — URL из Шага 1.

## Шаг 4. GitHub Pages

**Settings → Pages** → Source: `Deploy from a branch`, Branch: `main`, папка `/ (root)`.
Дашборд появится по адресу `https://ваш-логин.github.io/репозиторий/dashboard/`.

## Шаг 5. Первый запуск

Вкладка **Actions** → workflow «Sync Bitrix24 data (Victory)» → **Run workflow**.

**Первый запуск заберёт всю историю с 2026-01-01 — это может быть много записей
и занять несколько минут.** Дальнейшие запуски забирают только изменения и
работают быстро.

## Шаг 6. Проверьте сопоставление городов лидов — это важно

У сделок город определяется однозначно (через воронку). У лидов — по коду в
поле «Дополнительно об источнике» (например, `Victory_SPb` → Санкт-Петербург).
Таблица соответствий лежит в `scripts/city-map.js`, часть строк там —
**предположение** (я не мог проверить все города, только несколько по вашим
скриншотам).

После первого запуска откройте `data/meta.json` в репозитории и посмотрите поле
`unmatchedSiteCodes` — там перечислены все коды сайтов, которые скрипт **не
смог** сопоставить с городом (и сколько раз они встретились). Если там есть
что-то важное:

1. Откройте `scripts/city-map.js`
2. Добавьте/поправьте строку в объекте `CITY_MAP`, например:
   ```js
   novgorod: "Нижний Новгород",
   ```
   (ключ — код без подчёркиваний/пробелов в нижнем регистре)
3. Сохраните, задача при следующем запуске (или вручную через Run workflow)
   пересчитает данные правильно для новых записей.

Существующие уже сохранённые дни **не пересчитываются задним числом
автоматически** при правке таблицы — они обновятся только когда у
соответствующих лидов в Bitrix24 что-то изменится (это ограничение упрощённой
инкрементальной синхронизации, см. ниже). Если нужно пересчитать всю историю
заново после правки — удалите содержимое `data/months/` и `data/meta.json`
(кроме структуры) и запустите синхронизацию ещё раз как первый запуск.

## Как это устроено внутри (коротко)

- **Первый запуск**: тянет вообще все лиды/сделки источника с 2026-01-01
- **Каждый следующий запуск**: тянет только записи, которые **изменились**
  с последней синхронизации (по полю "дата изменения" в Bitrix24) — даже если
  созданы давно. Это нужно, чтобы у лида, который две недели считался
  "Не записан", а сегодня наконец дошёл до приёма, статус в дашборде
  обновился именно в дне его первого обращения, а не потерялся
- Каждый лид/сделка хранится как отдельная запись внутри своего дня и города
  (`data/months/2026-06.json` → `days["2026-06-15"].cities["Тольятти"].leads["1234"]`) —
  это позволяет пересчитывать показатели на лету при любом выборе периода/города
  на дашборде, без необходимости хранить готовые агрегаты на каждую комбинацию

## Траты на рекламу

В Bitrix24 таких данных нет, поэтому вводятся вручную в дашборде (кнопка
«Траты на рекламу» в шапке) — по городу и месяцу. Хранятся в браузере
(localStorage), поэтому:

- На другом устройстве/в другом браузере введённых трат видно не будет
- Кнопка экспорта в этом же окне выгружает текущие траты как JSON — этот
  текст можно сохранить в репозитории как файл `data/ad-spend.json` (через
  обычный редактор файлов на GitHub), тогда он станет общей базой для всех,
  кто открывает дашборд, а локальные правки в браузере будут накладываться
  поверх неё

## Ограничения, о которых стоит знать

- Обновление данных — строго по расписанию, не мгновенно (2 раза в сутки)
- Фильтр по датам — на уровне дня (не часов/минут)
- Новосибирск в сделках — 2 отдельные воронки («первичный ОП» и «вторичный ОП»),
  а в лидах разделить так же не получится (будет один общий Новосибирск)
- Если нужно чаще обновлять — добавьте ещё строк с `cron` в `.github/workflows/sync.yml`
